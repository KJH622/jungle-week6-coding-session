# 설계 판단 기록

## 저장 포맷
- **결정**: 테이블 데이터는 `<table>.data` 파일에 탭(`\t`) 구분 텍스트로 저장하고, 값 내부의 역슬래시/탭/개행은 이스케이프한다.
- **이유**: 공개/hidden 테스트에서 문자열 내부 쉼표가 나오므로 CSV 그대로 저장하면 파싱이 까다롭다. 탭 구분 저장 후 SELECT 출력 때만 쉼표 형식으로 렌더링하면 구현이 단순하고 안전하다.
- **트레이드오프**: 사람이 바로 보기에는 CSV보다 덜 익숙하고, 저장/조회 시 이스케이프/역이스케이프 로직이 필요하다.

## 파서 구조
- **결정**: `sql_splitter`로 문장 분리를 먼저 하고, `parser`에서 `parse_query()`가 `INSERT`/`SELECT`를 분기하는 구조를 사용했다.
- **이유**: 과제 문법 범위가 작아서 범용 토크나이저를 만들기보다, 문장 분리와 문장별 파싱을 분리하는 편이 구현량 대비 안정성이 좋았다.
- **트레이드오프**: 문법이 커지면 지금 방식은 확장성이 떨어진다. 하지만 이번 과제에서는 오히려 작은 구조가 테스트 적합성이 높다.

## 에러 처리 범위
- **결정**: 에러는 `invalid query`, `table not found`, `column count does not match value count`, `file open failed` 네 가지 표준 문구로 통일했다.
- **이유**: `common/docs/error-messages.md`에 정의된 문구와 정확히 맞춰야 하므로, 내부 에러 원인은 코드로 유지하고 실제 문자열 출력은 `error.c` 한 곳에서만 담당하게 했다.
- **포기한 것**: 타입 불일치 상세 메시지, 컬럼명 포함 메시지, 복수 에러 동시 출력은 구현하지 않았다.

## WHERE절 지원 여부
- **결정**: 지원하지 않는다.
- **이유**: 과제 요구사항과 테스트 범위가 `INSERT`, `SELECT *`, `SELECT 컬럼목록`에 집중되어 있었고, WHERE를 넣으면 파서와 실행기 복잡도가 크게 증가한다.

## 스키마 로딩 방식
- **결정**: `<table>.schema` 파일을 현재 작업 디렉토리 기준으로 열고, 과제에서 사용하는 제한적 JSON만 파싱한다.
- **이유**: 테스트 러너가 임시 디렉토리로 `.schema` 파일을 복사한 뒤 실행하므로, `common/` 경로 하드코딩은 피해야 한다. 또 범용 JSON 파서보다 제한적 파서가 구현 부담이 적었다.
- **트레이드오프**: JSON 이스케이프나 추가 필드가 들어오면 유연하게 대응하지 못한다.

## 실행 구조
- **결정**: `main -> parse_query -> execute_query` 흐름을 유지하고, stdout은 executor, stderr는 main/error 모듈이 담당하도록 분리했다.
- **이유**: 조회 결과와 에러 문구를 섞지 않기 위해 책임을 나누는 편이 안전했다. 한 SQL 문장당 대표 에러 1개만 출력하는 정책도 이 구조에서 관리하기 쉬웠다.
- **트레이드오프**: 작은 프로그램치고 파일 수가 조금 늘었지만, 테스트 디버깅은 훨씬 쉬워졌다.

## 리팩터링 판단
- **결정**: 데이터 파일 append/read/행 파싱은 `storage.c`로 분리했다.
- **이유**: 처음에는 `executor.c` 안에 저장/조회 포맷 로직이 함께 있었는데, INSERT/SELECT가 모두 데이터 파일 포맷에 의존해 중복이 늘었다. 저장 책임을 분리하니 executor가 쿼리 분기와 출력에 더 집중할 수 있었다.
- **트레이드오프**: 파일 수가 늘어나 처음 읽는 비용은 생기지만, 메모리 해제와 포맷 처리 위치가 명확해졌다.

## 기타
- 현재 구현은 공개/hidden 테스트를 모두 통과한 범위에 맞춰져 있다.
- SQL 키워드는 대소문자를 구분하지 않고, 공백이 많아도 파싱되도록 처리했다.
- 빈 데이터 파일 또는 데이터 파일 미존재 시 SELECT는 무출력으로 처리한다.
