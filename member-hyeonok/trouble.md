# 삽질 로그

## 이슈 1: 초기 저장소 구조와 실제 폴더명이 달라 보였던 문제
- **상황**: 처음 분석할 때 로컬 저장소에는 `hyeonok/`가 보였고, 사용자 요청에는 `member-hyeonok/`가 적혀 있어 작업 위치 판단이 애매했다.
- **원인**: `git fetch` 전 상태와 원격 최신 상태가 달랐고, 개인 폴더명 변경 커밋이 아직 로컬에 반영되지 않은 상태였다.
- **AI가 준 답**: 현재 보이는 로컬 구조 기준으로 `hyeonok/`를 사용하자고 안내했다.
- **실제 해결**: `git fetch origin` 후 저장소를 다시 확인해서 실제 최신 구조가 `member-hyeonok/`임을 확인했고, 이후 작업은 모두 그 아래에서 진행했다.
- **배운 것**: 과제 규칙과 저장소 구조가 충돌해 보이면 먼저 원격 최신 상태를 확인해야 한다.

## 이슈 2: 문자열 값 trim으로 공백이 손실될 수 있었던 문제
- **상황**: INSERT 파서를 만들 때 문자열 값도 일반 토큰처럼 trim 처리하면 될 것처럼 보였다.
- **원인**: 공백 제거 헬퍼를 재사용하는 편이 편했지만, `'김 민준'`, `'컴퓨터 공학과'` 같은 값은 공백 자체가 데이터였다.
- **AI가 준 답**: 문자열 값도 기존 토큰 복사 함수를 써서 trim하는 방향이 처음 들어왔다.
- **실제 해결**: 문자열 값 전용으로 `copy_exact_range()`를 두고, 작은따옴표 내부 값은 trim 없이 그대로 저장하도록 수정했다.
- **배운 것**: 파서에서 “문자열 리터럴”은 일반 식별자 토큰과 다르게 다뤄야 한다.

## 이슈 3: 데이터 저장 포맷을 CSV로 두면 쉼표 값에서 깨지는 문제
- **상황**: 처음에는 `.csv`나 쉼표 저장이 단순해 보여서 고민했다.
- **원인**: hidden 테스트에 문자열 내부 쉼표가 들어가는 케이스가 있었고, CSV 이스케이프를 제대로 지원하지 않으면 저장/조회가 깨진다.
- **AI가 준 답**: 사람이 읽기 쉬운 CSV를 추천하는 방향이 먼저 나왔다.
- **실제 해결**: 저장 포맷은 탭 구분 + 이스케이프 텍스트로 두고, SELECT 출력 때만 쉼표로 렌더링했다.
- **배운 것**: “출력 포맷”과 “내부 저장 포맷”은 분리하는 편이 구현 안전성이 높다.

## 이슈 4: 에러 문자열이 여러 파일에 흩어져 오타 위험이 있었던 문제
- **상황**: main에서 직접 `fprintf(stderr, "ERROR: ...")`를 여러 번 쓰다 보니, 제출 직전에는 문구 일관성이 불안했다.
- **원인**: parser/executor 에러를 문자열로 바로 찍는 구조는 작은 오타나 공백 차이에도 취약하다.
- **AI가 준 답**: 에러를 코드로만 전달하고, 상위에서 한 번에 출력하도록 공통 모듈로 빼자는 제안을 했다.
- **실제 해결**: `error.c/.h`를 만들어 `AppError`와 표준 메시지 매핑을 한곳에 모았다. 이후 stderr 출력은 `print_app_error()`만 사용하게 정리했다.
- **배운 것**: 테스트가 문자열 diff에 민감할수록 에러 메시지는 중앙 집중형으로 관리하는 편이 낫다.

## 이슈 5: executor 안에 저장/조회 포맷 로직이 몰려 읽기 어려웠던 문제
- **상황**: INSERT 저장과 SELECT 조회가 모두 `executor.c` 안에 있다 보니, 데이터 파일 포맷 함수까지 뒤섞여 책임이 흐려졌다.
- **원인**: 기능을 빠르게 붙이는 과정에서 데이터 파일 append/read/역직렬화가 실행기 안으로 들어갔다.
- **AI가 준 답**: 동작을 바꾸지 않는 선에서 `storage` 모듈로 분리하자는 리팩터링 방향을 제안했다.
- **실제 해결**: `.data` 경로 생성, append, 전체 읽기, 행 파싱을 `storage.c/.h`로 옮겼고, `executor`는 쿼리 분기와 출력 중심으로 단순화했다.
- **배운 것**: 기능이 모두 맞은 뒤 책임 분리를 하면 테스트를 유지하면서도 구조를 크게 개선할 수 있다.

## 이슈 6: 테스트 실행 중 임시 디렉토리 재사용으로 결과가 헷갈린 문제
- **상황**: 종료코드 검증 중에 이전 테스트에서 만들어진 `users.data`가 남아 있어 SELECT 출력이 예상보다 많아 보였다.
- **원인**: 같은 임시 디렉토리를 여러 번 재사용하면서 기존 데이터 파일이 누적됐다.
- **AI가 준 답**: 종료코드 검증 자체는 맞지만, 깨끗한 새 작업 디렉토리에서 다시 확인하자는 방향을 제안했다.
- **실제 해결**: 테스트 케이스별로 새 임시 디렉토리를 만들고 schema만 다시 복사해서 재검증했다.
- **배운 것**: 파일 기반 과제에서는 테스트 격리 환경을 직접 의식해야 결과를 정확히 해석할 수 있다.
