# AI 활용 이력

## 사용 도구
- Codex (GPT-5 기반 코딩 에이전트)

## 프롬프트 기록

### 1단계: 뼈대 구축
> 기존 저장소 구조와 `common/` 테스트 규칙을 먼저 분석하고, `member-hyeonok/` 아래에서만 작업 가능한 SQL Processor의 초기 프로젝트 뼈대, Makefile, `main -> SQL 파일 읽기 -> 세미콜론 분리 -> parser/executor 인터페이스` 흐름을 C로 만들어 달라. 핵심 함수에는 한글 주석을 달고, 아직은 구조 안정화에 집중하라.

**AI 응답 요약**: `sql_splitter`, `parser`, `executor`, `util`, `main`으로 나눈 기본 구조와 Makefile, SQL 파일 전체 읽기/문장 분리 흐름을 제안하고 구현했다.
**내 판단**: 구조 분리 방향은 그대로 채택했다. 과제 규모에 비해 과한 추상화는 피하고, 현재 작업 디렉토리 기준 파일 처리라는 제약은 문서 분석 결과에 맞춰 유지했다.

### 2단계: 파서 구현
> `INSERT INTO <table> (...) VALUES (...)`와 `SELECT * FROM <table>`, `SELECT col1, col2 FROM <table>`를 실제 AST로 채우는 파서를 구현하라. SQL 키워드는 대소문자를 구분하지 않고, 공백이 많아도 파싱 가능해야 하며, 문법 오류는 이후 `invalid query`로 연결 가능한 형태로 반환하라.

**AI 응답 요약**: `SqlStatement`, `InsertStatement`, `SelectStatement`, `SqlValue`를 보강하고 `parse_query()` 중심으로 INSERT/SELECT 파서를 구현했다.
**내 판단**: AST 구조 확장은 채택했다. 다만 문자열 값은 trim하지 않고 원문 그대로 보존하도록 수정해 값 내부 공백이 깨지지 않게 했다.

### 3단계: 스키마 로딩
> `<table>.schema` JSON 파일을 현재 작업 디렉토리 기준으로 읽어 `Schema` 구조체로 로딩하라. 범용 JSON 파서는 만들지 말고, 과제에서 사용하는 `"table"`과 `"columns"` 형식만 안정적으로 읽게 하라.

**AI 응답 요약**: 제한적 JSON 파서를 가진 `schema.c/.h`를 추가하고, `SCHEMA_LOAD_SUCCESS / TABLE_NOT_FOUND / FILE_ERROR` 반환 규약을 제안했다.
**내 판단**: 현재 테스트 구조와 잘 맞아서 그대로 채택했다. 절대경로/`common/` 의존 없이 `<table>.schema`만 조합해서 찾는 방식이 중요하다고 판단했다.

### 4단계: INSERT/SELECT 실행기 구현
> 파서 결과를 바탕으로 INSERT는 데이터 파일에 append 저장하고, SELECT는 데이터 파일을 읽어 헤더와 행을 쉼표 형식으로 출력하라. 빈 테이블, 잘못된 컬럼, 없는 테이블, 에러 출력 규칙까지 `common` 명세에 맞춰라.

**AI 응답 요약**: INSERT는 스키마 순서 기준 `.data` append, SELECT는 `.data` 읽기와 컬럼 선택 출력, 대표 에러 코드 반환 구조를 구현했다.
**내 판단**: 전반은 채택했다. 데이터 저장 포맷은 쉼표 대신 탭 구분 + 이스케이프 방식으로 유지해 값 내부 쉼표 hidden 테스트에 대비했다.

### 5단계: 종료코드/에러 정책 정리
> 여러 SQL 문장을 계속 실행하면서도, 하나라도 실패하면 최종 종료코드 1을 반환하도록 main 흐름과 에러 출력을 정리하라. 표준 에러 문구는 정확히 네 개만 사용하고 stderr로만 출력되게 하라.

**AI 응답 요약**: `run_single_statement()`와 공통 `error.c`를 두어 parser/executor는 코드만 반환하고, 실제 stderr 문자열 출력은 상위에서만 하도록 정리했다.
**내 판단**: 그대로 채택했다. 문자열이 여러 곳에 흩어져 있으면 오타가 날 가능성이 커서, 제출 직전에는 공통 에러 모듈이 더 안전했다.

### 6단계: 구조 리팩터링 및 제출 문서 정리
> 동작을 바꾸지 않고 구조와 메모리 관리를 정리하라. 특히 데이터 파일 처리 책임을 분리하고, 제출 전 확인 문서와 체크리스트를 실제 구현에 맞게 업데이트하라.

**AI 응답 요약**: `storage.c/.h`로 데이터 파일 append/read/행 파싱을 분리하고, `build-and-run.md`, `submission-checklist.md`를 현재 구현 기준으로 정리했다.
**내 판단**: 채택했다. 기능이 모두 들어간 뒤에 책임 분리를 적용하니 테스트를 유지하면서 구조를 더 읽기 쉽게 만들 수 있었다.

## 사용 방식에 대한 메모

- 각 단계마다 먼저 로컬 저장소와 `common` 문서를 읽고, 그 근거 위에서 구현하도록 요청했다.
- 구현 단계에서는 “큰 구조 변경보다 현재 테스트 적합성 우선” 원칙을 계속 유지했다.
- 문서화 단계에서는 실제 구현 범위를 벗어나는 설명을 넣지 않도록 별도 제출 체크리스트를 만들었다.
