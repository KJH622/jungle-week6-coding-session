# SQL Processor Flow

## 1. 프로그램 시작
1. `./sql_processor <sql_file>` 인자를 검증한다.
2. 인자 오류(없음/과다) 또는 파일 열기 실패 시 `ERROR: file open failed`를 `stderr`에 출력하고 종료코드 `1`로 종료한다.

## 2. 초기화
1. 필수 스키마(`users`, `products`)를 로드한다.
2. 스토리지를 초기화한다. 현재 구현은 CSV 기반 저장소를 사용한다.
3. 초기화 실패 시 에러를 출력하고 종료코드 `1`로 종료한다.

## 3. 입력 처리
1. SQL 파일 내용을 읽는다.
2. 빈 줄/공백을 정리한다.
3. `;` 기준으로 SQL 문장을 분리한다.

## 4. 문장별 실행 루프
1. 각 SQL 문을 파싱한다. 지원 범위는 `INSERT`, `SELECT`다.
2. 파싱 실패 시 `ERROR: invalid query`를 출력하고 다음 문장으로 진행한다.

## 5. INSERT 처리
1. 테이블 존재 여부를 검증한다. 없으면 `ERROR: table not found`.
2. 컬럼/값 개수를 검증한다. 불일치하면 `ERROR: column count does not match value count`.
3. 검증 통과 시 레코드를 파일에 append한다.
4. 성공 시 출력은 없다.

## 6. SELECT 처리
1. 테이블/컬럼 유효성을 검증한다. 잘못된 컬럼은 `ERROR: invalid query`.
2. 데이터 파일을 조회한다.
3. 결과가 있으면 `헤더 + 데이터 행`을 CSV로 출력한다.
4. 결과가 없으면 출력하지 않는다.

## 7. 에러 및 종료 처리
1. 한 SQL 문장에서는 대표 에러 1개만 출력한다.
2. 에러가 발생해도 다음 SQL 문장을 계속 실행한다.
3. 전체 실행 중 에러가 1건 이상이면 종료코드 `1`, 없으면 `0`으로 종료한다.

## 8. 확장 포인트
1. 스토리지 인터페이스를 유지하면 백엔드를 교체할 수 있다.
2. CSV 구현을 SQLite 구현으로 교체해 동시성/확장 요구에 대응할 수 있다.
